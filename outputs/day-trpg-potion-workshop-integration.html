오늘은 기존에 챕터6까지 구현했던 C++ TRPG 프로젝트에 Step7 포션 제작소 기능을 추가하고, 지금까지 만들었던 기능들과 통합하는 작업을 진행했다.

프로젝트 전체 점검

먼저 현재 C++ TRPG 프로젝트의 파일과 코드를 전체적으로 살펴봤다.

프로젝트가 정상적으로 빌드되는지 확인했고, 메뉴 입력 처리, 직업별 공격 함수, 인벤토리 제한 등 앞으로 개선해야 할 부분도 함께 확인했다.

아직 완성된 프로젝트는 아니지만, 지금까지 만든 기능들이 서로 어떻게 연결되어 있는지 다시 점검하는 시간이 되었다.

포션 제작소 설계

포션 제작소를 구현하기 위해 다음 구조를 생각했다.

  • PotionRecipe 구조체 만들기
  • AlchemyWorkshop 클래스 만들기
  • vector<PotionRecipe>에 레시피 저장하기
  • 전체 레시피 출력 기능 만들기
  • 포션 이름으로 검색하는 기능 만들기
  • 재료 이름으로 검색하는 기능 만들기
  • 포션 제작소 메뉴 만들기
  • 게임 메인 메뉴와 제작소 연결하기

포션 제작소는 단순히 포션 정보를 출력하는 기능이 아니라, 레시피를 저장하고 검색하는 작은 시스템처럼 동작하도록 만들었다.

PotionRecipe 구조체

먼저 포션 레시피 정보를 저장하기 위해 PotionRecipe 구조체를 만들었다.

포션 이름, 첫 번째 재료, 첫 번째 재료 개수, 두 번째 재료, 두 번째 재료 개수 같은 정보를 하나로 묶어 관리할 수 있도록 했다. 

struct PotionRecipe
{
    string potionName;
    string firstIngredient;
    int firstIngredientCount;
    string secondIngredient;
    int secondIngredientCount;
};

구조체를 사용하니 포션 하나에 필요한 정보를 따로따로 관리하지 않고, 하나의 단위로 묶어서 다룰 수 있었다.

포션 레시피 생성과 저장

HP 포션과 스태미나 포션 정보를 작성하고, 만든 레시피를 vector에 저장했다. 

PotionRecipe HPpotion;
PotionRecipe STMpotion;

작성한 레시피는 push_back()을 사용해서 벡터에 넣었다. 

recipes.push_back(HPpotion);
recipes.push_back(STMpotion);

push_back()은 괄호 안의 값을 벡터 맨 뒤에 추가하는 기능이다. 이전에 배웠던 vector 개념을 다시 복습할 수 있었다.

전체 레시피 출력

전체 레시피를 출력할 때는 범위 기반 for문을 사용했다. 

for (const PotionRecipe& recipe : recipes)
{
    // 레시피 출력
}

여기서 const PotionRecipe&를 사용한 이유는 레시피를 복사하지 않고 참조하되, 출력만 하고 수정하지 않기 위해서이다.

각 레시피의 포션 이름, 재료 이름, 필요한 개수를 출력하도록 ShowAllRecipes() 함수를 완성했다.

포션 이름 검색

포션 이름으로 레시피를 찾는 SearchByName() 함수를 만들었다.

구현 흐름은 다음과 같다.

  • 모든 레시피를 반복한다.
  • 입력한 포션 이름과 레시피의 포션 이름을 비교한다.
  • 이름이 같으면 해당 레시피를 출력한다.
  • bool found로 검색 성공 여부를 기억한다.
  • 결과가 없으면 찾을 수 없습니다.를 출력한다.
bool found = false;

for (const PotionRecipe& recipe : recipes)
{
    if (recipe.potionName == searchName)
    {
        // 레시피 출력
        found = true;
    }
}

if (!found)
{
    cout << "찾을 수 없습니다." << endl;
}

여기서 found는 포션 이름을 저장하는 변수가 아니라, 일치하는 레시피를 한 번이라도 찾았는지 기억하는 표시 변수이다.

재료 검색

재료 이름으로 레시피를 찾는 SearchByIngredient() 함수도 구현했다.

한 레시피에는 첫 번째 재료와 두 번째 재료가 있기 때문에, 두 재료 모두 검색 대상에 포함해야 했다. 

if (recipe.firstIngredient == searchIngredient ||
    recipe.secondIngredient == searchIngredient)
{
    // 레시피 출력
    foundCount++;
}

||는 논리 OR 연산자이다. 즉, 첫 번째 재료가 같거나 두 번째 재료가 같으면 검색 결과로 인정한다.

검색 결과 개수는 foundCount로 누적했다. 반복문이 끝난 뒤 foundCount가 0이면 검색 결과가 없다는 문장을 출력했다.

포션 제작소 메뉴

제작소 안에서 사용할 수 있는 RunMenu() 함수를 만들었다. 

=== 포션 제작소 ===
1. 전체 레시피 보기
2. 포션 이름으로 검색
3. 재료로 검색
0. 돌아가기

메뉴 선택에 따라 다음 기능이 실행되도록 했다.

  • case 1: 전체 레시피 출력
  • case 2: 포션 이름 검색
  • case 3: 재료 검색
  • case 0: return으로 메인 메뉴 복귀

return을 사용하면 제작소 메뉴 함수가 종료되고, 다시 게임의 메인 메뉴 흐름으로 돌아갈 수 있다.

메인 메뉴와 제작소 연결

main()에서 제작소 객체를 만들었다. 

AlchemyWorkshop workshop;

그리고 메인 메뉴의 3번 선택지에 제작소를 연결했다. 

case 3:
    workshop.RunMenu();
    break;

이 과정을 통해 기존 게임 흐름에 새로운 기능을 자연스럽게 추가하는 방법을 배웠다.

공백이 포함된 검색어 문제

포션 이름을 검색할 때 cin >> searchName을 사용하면 문제가 생겼다.

예를 들어 HP 포션을 입력하면 HP까지만 읽고, 포션은 입력 버퍼에 남게 된다.

이 문제를 해결하기 위해 getline(cin >> ws, searchName)을 사용했다. 

getline(cin >> ws, searchName);

getline()은 공백을 포함한 한 줄 전체를 입력받을 수 있다. cin >> ws는 앞에 남아 있는 공백 문자를 제거해준다.

메뉴 입력 실패 처리

메뉴 선택에 숫자가 아닌 문자를 입력했을 때 cin이 실패 상태에 빠지는 원리도 살펴봤다.

이전에 능력치 입력 검사에서 사용했던 것처럼, 메뉴 입력에서도 cin.fail(), cin.clear(), cin.ignore()를 활용할 수 있다는 것을 다시 확인했다.

직업별 공격 함수 연결

전사를 선택했는데도 검을 휘두른다! 같은 직업별 공격 문구가 출력되지 않는 문제가 있었다.

원인을 확인해보니 attack() 함수를 선언하고 구현했지만, 실제 게임 흐름에서 호출하지 않았기 때문이었다.

전직 후 다음 코드를 호출하도록 연결했다. 

player->attack();

이를 통해 부모 클래스 포인터인 player를 통해 호출해도, 실제 선택한 직업 클래스의 attack()이 실행되는 다형성을 확인할 수 있었다.

복습한 C++ 개념

오늘은 구현뿐만 아니라 지금까지 배운 C++ 개념도 다시 복습했다.

  • int&는 포인터가 아니라 레퍼런스이다.
  • 레퍼런스는 원본 변수의 별명이다.
  • !는 논리 부정 연산자이다.
  • !true는 false이다.
  • foundCount를 반복문 밖에서 선언해야 검색 결과 개수를 누적할 수 있다.
  • ||는 논리 OR, 즉 "또는"이라는 의미이다.
  • push_back()은 벡터의 맨 뒤에 값을 추가한다.
  • 범위 기반 for문은 컨테이너의 원소를 하나씩 꺼내 반복한다.
  • const는 값을 읽기만 하고 수정하지 못하게 보호한다.
  • found는 검색 성공 여부를 기억하는 bool 변수이다.

AI를 활용한 공부 방식

오늘은 AI가 없으면 코딩하기 어려울 것 같다는 고민도 했다.

앞으로는 완성 코드를 바로 받기보다 다음 흐름으로 공부하기로 했다.

  1. 요구사항을 말로 정리한다.
  2. 작은 단계로 나눈다.
  3. 직접 코드를 작성한다.
  4. 오류 원인을 먼저 예상해본다.
  5. 필요할 때 힌트를 받는다.
  6. 완성 후 자기 말로 다시 설명한다.

이 방식으로 공부하면 단순히 코드를 복사하는 것이 아니라, 내가 직접 구현 과정을 이해하면서 성장할 수 있을 것 같다.

느낀 점

처음에는 Text RPG가 간단한 프로젝트라고 생각했다.

하지만 직접 만들다 보니 캐릭터 생성, 능력치 입력 검사, 강화 메뉴, 직업 선택, 전투, 아이템, 인벤토리, 포션 제작소까지 생각보다 많은 코딩 능력이 필요하다는 것을 몸으로 느꼈다.

특히 하나의 기능만 따로 만들 때보다, 만든 기능을 기존 게임 흐름과 연결하는 과정이 더 어렵게 느껴졌다.

작성한 코드를 보며 복습하다 보니 분명 어려웠던 챕터들이 있었다. 그 부분은 앞으로 TIL을 다시 보면서 복습하고, 부족한 능력을 조금씩 키워야겠다고 느꼈다.

오늘 작업을 통해 단순히 코드를 작성하는 것보다 기능을 설계하고, 연결하고, 다시 점검하는 과정이 중요하다는 것을 알게 되었다.

어려웠던 점

가장 어려웠던 점은 새로 만든 포션 제작소 기능을 기존 메인 메뉴 흐름에 자연스럽게 연결하는 것이었다.

포션 제작소 안에서도 별도의 메뉴가 있고, 그 메뉴에서 다시 메인 메뉴로 돌아와야 했기 때문에 함수의 종료 시점과 return, break의 차이를 신경 써야 했다.

또한 공백이 포함된 문자열 입력도 헷갈렸다. cin >> searchName은 공백 전까지만 입력받기 때문에 HP 포션 같은 이름을 제대로 처리하지 못했다.

이를 해결하기 위해 getline(cin >> ws, searchName)을 사용했고, 입력 버퍼와 공백 처리에 대해 조금 더 이해할 수 있었다.

파일을 .h와 .cpp로 분리하는 것도 아직 익숙하지 않았다. 어떤 선언을 헤더에 두고, 어떤 구현을 cpp에 둘지 판단하는 연습이 더 필요하다고 느꼈다.
